# 记忆链路升级优化计划

> 目标：把"能演示"的记忆系统做成"能日常用"——先解决**准不准**，再解决**脏不脏**，最后解决**控不控得住**。
> 原则：规则版先做实，语义检索后置；不引入向量库等重依赖，本地几百条记忆内存内即可算完。

---

## 一、现状与链路

当前记忆链路的骨架是完整且设计得当的：

```
用户消息 ──► extractCandidatesWithModel（每条消息调一次模型，只看当前这一条）
                │
                ▼
        候选区（仅存前端 React state，上限 6 条）
                │ 用户点"审核 N"（一键全量提交）
                ▼
        /api/memory/commit ──► raw/YYYY-MM-DD.md（审计日志）+ index.json ──► 渲染 memory.md
                                                            │
下一条用户消息 ──► selectRelevantMemories（关键词打分取 top 8）──► 注入 <long_term_memory>
```

**值得保留的骨架（重构时勿丢）**：候选先审后写、`raw/` append-only 审计日志、注入时的 `<long_term_memory>` 不可信内容包裹、strict/宽松双档检索、写入互斥锁。以下所有优化都是在这个骨架上把每一环做实。

### 三个硬伤（决定改动顺序）

| # | 问题 | 位置 | 后果 |
|---|---|---|---|
| 1 | 中文检索基本失效 | `server/memory.mjs:10` | 中文无空格，整句被当成一个关键词，`includes` 几乎不命中，检索退化成按 level 固定注入同样 8 条 |
| 2 | 无去重/更新机制 | `server/memory.mjs:58`、`server/app.mjs:475` | 同一件事说三次存三条；偏好变更后新旧矛盾记忆并存注入 |
| 3 | 无遗忘机制 | `server/app.mjs:485` | 记忆只进不出，organize 是空操作，错误/过期记忆无出口 |

---

## 二、阶段一：让检索真正工作（修正确性）· 约 0.5–1 天

收益最高的一步，改动集中在 `server/memory.mjs` 的 `selectRelevantMemories`。

**1.1 中文分词**
把 `memory.mjs:10` 的 `split(/[^\p{L}\p{N}]+/u)` 换成对 CJK 友好的分词。推荐 `Intl.Segmenter('zh', { granularity: 'word' })`（Node 18+ 内置，零依赖）；不想依赖 ICU 则用 CJK 二字滑窗（bigram）。记忆内容侧用同样方式建词集，两边求交集算命中数。

**1.2 分数阈值 + 分桶注入**
现在 level 基础分让 0 命中的条目也能入选。改成两个桶：
- **常驻桶**：最多 3 条 `high` 级偏好，无条件注入（如"回复用中文"这类全局规则）。
- **检索桶**：最多 5 条，**必须 keywordHits ≥ 1** 才入选。

打分公式建议：`score = 2*keywordHits + recencyBoost(updatedAt) + 0.5*levelWeight`，keyword 为主信号，level 只做微调。

**1.3 时间衰减**
把 `updatedAt` 纳入打分，`recencyBoost` 在 ~90 天内从 1 衰减到 0，让新记忆天然更易召回。

**1.4 截断规则修复**
- `memory.mjs:121` 的 96 字符硬截断对中文长偏好会拦腰截断，放宽到 ~200 或改成 token 感知。
- `memory.mjs:105` 的"不足 8 字符丢弃"会误杀"回复用中文"这类 5 字偏好，降到 4 或对 CJK 单独放宽。

**完成标准**：发"我喜欢深色主题"后再问"配色建议"，能召回这条；无关记忆不再被固定注入。

---

## 三、阶段二：去重与更新（upsert 语义）· 约 1 天

解决"同一件事说三次存三条"和"偏好变更时新旧矛盾并存"。

**2.1 抽取时带上下文和已有记忆**
改 `memory.mjs:58` `extractCandidatesWithModel` 的 prompt：喂入最近 3–4 轮对话（不只当前一条）+ 现有活跃记忆的 `{id, content}` 列表。

**2.2 让模型输出动作而非只会新增**
输出结构从 `candidates[]` 升级为 `actions[]`，每项 `{op: "add"|"update"|"noop", targetId?, content, type, level}`。"以后改用深色主题"会被识别为 `update` 掉旧的浅色记忆，而非新增矛盾条目。

**2.3 commit 做语义合并**
改 `app.mjs:475` 的合并逻辑：`update` 走 targetId 覆盖内容并 bump `updatedAt`；`add` 再加一层内容归一化哈希做兜底去重。

**完成标准**：重复表达同一偏好不产生重复条目；偏好变更后旧条目被更新/失效，不会两条一起注入。

---

## 四、阶段三：候选持久化 + 逐条审核 · 约 1 天

解决"候选刷新即失、第 7 条被静默丢弃、审核名不副实"。

**3.1 候选落盘**
数据模型里 `status: "candidate"` 早已存在，服务端却从未写入。让抽取结果直接以 candidate 状态写进 `index.json`；前端从 memoryState 读候选，不再依赖易失的 React state（去掉 `src/hooks/useWorkbenchState.ts:131` 的 `.slice(0, 6)`）。

**3.2 逐条审核 UI**
把 `src/components/MemoryPanel.tsx` 的"审核 N / 一键全提交"改成每条候选可**接受 / 拒绝 / 编辑**（改内容、等级、类型后再提交）。这是候选区的核心产品价值所在。

**完成标准**：候选跨刷新不丢；能单条改内容和等级后入库；能拒绝单条不影响其他。

---

## 五、阶段四：遗忘与真整理 · 约 1 天

解决"记忆只进不出、organize 是空操作"。

**4.1 单条记忆的编辑/禁用/删除**
后端加 `PUT/DELETE /api/memory/:id`（走同一把写锁），前端每张 MemoryCard 加对应入口。让错误、过期、矛盾的记忆有出口。

**4.2 organize 接模型做真整理**
`app.mjs:485` 现在只是把所有条目重标 active 再重渲染。改成一次模型调用：合并相似项、消解矛盾、淘汰过期，返回整理后的 index 供用户确认（建议整理结果先进预览/候选，别直接覆盖）。

**完成标准**：能禁用/删除单条记忆；点"整理"后重复项被合并、矛盾被消解，且用户可预览确认再落盘。

---

## 六、阶段五（中期）：语义检索 · 约 2–3 天

规则版做实之后再上，不必上向量库。

**5.1 embedding 缓存**：调所选 provider 的 embedding API，把向量直接存在 `index.json` 每条记忆上（写入/更新时算一次）。
**5.2 内存余弦相似度**：检索时对 query 向量与几百条记忆向量在内存里算余弦，毫秒级，零存储依赖。
**5.3 关键词打分降级**：阶段一的关键词逻辑保留，作为 embedding 不可用（未配 key / 调用失败）时的 fallback——所以阶段一的工作不会白做。

**完成标准**：语义相近但不同词的表达也能召回（"配色偏好"能召回"深色主题"）；embedding 失效时自动降级到关键词，不报错。

---

## 七、顺带清理（低成本，随手做）

- **删死代码**：规则版抽取 `generateCandidatesFromMessages` 和端点 `app.mjs:453` 前端从未调用，与模型版并存两套；删掉，或明确降级为模型不可用时的 fallback。
- **memory.md 加头注释**：它是 index.json 的派生物，每次 commit/organize 重新生成，用户手改会被静默覆盖。文件头加一行"自动生成，请勿直接编辑"。
- **provenance 展开**：`memoryRefs` 已记录每条回复引用了哪些记忆，`src/components/ChatPanel.tsx:137` 却只显示"引用 N 条"。加个展开看具体哪几条，让记忆作用可解释。

## 八、推荐排期

| 时段 | 内容 | 理由 |
|---|---|---|
| 第一周 | 阶段一 + 二 | 直接决定"准不准、脏不脏"，是 demo → 可日常用的分水岭，且互不阻塞可并行 |
| 第二周 | 阶段三 + 四 | 补齐"可控性"（候选持久化 + 遗忘） |
| 之后 | 阶段五 | 等前四步稳定、有真实使用数据后再上，避免过早优化 |

清理项穿插在任意阶段随手完成。

## 九、验收清单（整体）

- [ ] 中文消息能按语义召回相关记忆，无关记忆不再固定注入
- [ ] 重复偏好不产生重复条目，偏好变更能更新/失效旧记忆
- [ ] 候选跨刷新不丢，可逐条接受/拒绝/编辑
- [ ] 单条记忆可编辑/禁用/删除，organize 是真合并而非空操作
- [ ] （中期）语义检索可用，且 embedding 失效时自动降级
