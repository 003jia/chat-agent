# 记忆链路升级优化计划

> 更新时间：2026-07-15
>
> 当前状态：阶段一至五已完成代码实现；语义检索需在模型设置中配置 Embedding 模型后启用。
> 目标：把“能演示”的记忆系统做成“能日常用”——先解决准不准，再解决脏不脏，最后解决控不控得住。

## 一、当前架构

```text
用户消息
  ├─► 关键词检索（CJK 分词、命中阈值、时间衰减）
  ├─► 可选语义检索（Embedding 缓存、内存余弦相似度）
  └─► 相关记忆分桶（常驻最多 3 条 + 相关最多 5 条）
          │
          ▼
  注入 <long_term_memory> 不可信内容块
          │
          ▼
      模型流式回复 ──► 先返回 message.done
          │
          ▼
  后台抽取 actions（add / update / noop）
          │
          ▼
  candidate 写入 index.json ──► 用户逐条接受 / 编辑 / 拒绝
          │
          ▼
  commit ──► raw/YYYY-MM-DD.md 审计日志
         └─► index.json 主数据 ──► 派生 memory.md
```

继续保留的核心原则：

- 候选先审后写，不自动污染长期记忆。
- `raw/` 保持 append-only，作为可追溯审计日志。
- `index.json` 是主数据，`memory.md` 是自动生成的可读派生文件。
- 长期记忆、搜索结果和用户文本均按不可信内容处理。
- strict / 宽松双档检索继续生效。
- 所有记忆写操作使用同一把互斥锁。
- Embedding 不可用时自动降级到关键词检索，不阻断聊天。

## 二、阶段一：让检索真正工作（已完成）

### 已实现

- 使用 `Intl.Segmenter`、CJK 单字和二字滑窗共同处理中文分词。
- 常驻桶最多加载 3 条 `high` 级用户偏好。
- 相关桶要求至少命中一个关键词，不再仅凭等级固定注入。
- 相关性分数以关键词为主，并加入等级微调和 90 天时间衰减。
- 宽松检索最多 8 条，strict 检索最多 5 条。
- 记忆最短长度降为 4 字符，摘要上限提高到 200 字符。
- 每次引用会更新 `accessCount` 和 `lastAccessedAt`。

### 关键实现

- `server/memory.mjs`：`tokenizeMemoryText`、`selectRelevantMemories`、`updateMemoryAccess`
- `server/constants.mjs`：检索数量、最小命中、摘要和时间衰减常量
- `server/memory.test.mjs`：中文召回、无关记忆过滤、常驻桶和 strict 上限测试

## 三、阶段二：去重与更新（已完成）

### 已实现

- 候选抽取输入最近 3–4 轮对话以及已有活跃记忆。
- 模型输出从单纯 `candidates[]` 升级为 `actions[]`。
- 支持 `add`、`update`、`noop`，整理流程额外支持 `disable`。
- `update` 使用 `targetId` 更新原记忆，不创建互相矛盾的新条目。
- `add` 使用归一化内容哈希做兜底去重。
- 内容或类型发生变化时会重新生成 hash、keywords，并使旧 Embedding 缓存失效。

### 关键实现

- `server/memory.mjs`：`extractCandidatesWithModel`、`mergeCandidateMemories`、`commitMemoryItems`
- `server/memory.test.mjs`：重复候选转 update、更新目标记忆、禁用目标记忆测试

## 四、阶段三：候选持久化和逐条审核（已完成）

### 已实现

- 抽取结果以 `status: "candidate"` 写入 `index.json`，页面刷新后不会丢失。
- 流式回复先发送 `message.done`，候选抽取完成后再发送 `memory.candidates`。
- 每条候选都支持接受、编辑和拒绝。
- 接受单条候选不会影响其他候选。
- 右侧面板展示前 5 条以控制密度，详细抽屉展示全部候选，不再静默丢弃。
- assistant 消息记录 `candidateMemoryIds`，候选和来源回复可关联。

### 关键实现

- `server/app.mjs`：后台候选抽取、候选持久化、commit 和 patch 路由
- `src/components/MemoryPanel.tsx`：逐条接受 / 编辑 / 拒绝
- `server/app.test.mjs`：候选持久化、拒绝和并发提交测试

## 五、阶段四：遗忘与真整理（已完成）

### 已实现

- 单条记忆支持编辑、禁用和永久删除，全部经过写锁。
- “整理记忆”调用模型识别重复、冲突和过期条目。
- 模型整理结果先写成候选，用户确认后才修改活跃记忆。
- 没有可用模型或模型整理失败时，使用本地近重复检测生成禁用候选。
- `memory.md` 文件头明确标注为自动生成文件，避免用户手改后被覆盖。
- 聊天气泡可展开查看实际引用的记忆内容。

### 关键实现

- `server/app.mjs`：`PATCH/DELETE /api/memory/:memoryId`、`POST /api/memory/organize`
- `server/memory.mjs`：`organizeMemoryWithModel`、`createLocalOrganizeCandidates`
- `src/components/ChatPanel.tsx`：记忆引用展开
- `server/app.test.mjs`：整理结果进入候选区、不直接覆盖活跃记忆

## 六、阶段五：可选语义检索（已完成）

### 启用方式

在模型设置中填写 `Embedding 模型（可选）`。OpenAI 或支持 OpenAI-compatible `/embeddings` 接口的供应商可以填写对应模型名，例如 `text-embedding-3-small`。留空时只使用关键词检索；Anthropic 当前不启用此入口。

### 已实现

- 记忆写入或更新后，在后台批量生成 Embedding 缓存。
- 缓存直接保存在 `index.json`：`embedding`、`embeddingModel`、`embeddingHash`、`embeddingUpdatedAt`。
- 首次语义查询会为尚未缓存的活跃记忆批量补齐向量。
- 查询向量与记忆向量在内存中计算余弦相似度，不引入向量数据库。
- 关键词命中或语义相似度达到阈值均可进入候选集合。
- 混合分数继续保留关键词、时间和等级信号，语义相似度不会独占排序。
- 模型切换或记忆内容变化后，旧缓存会失效并重新生成。
- Embedding 请求失败时记录脱敏错误，并自动回退关键词检索。
- “测试连接”在配置 Embedding 模型后会同时验证聊天和 Embedding 接口。

### 关键实现

- `server/model.mjs`：`callEmbeddings`
- `server/memory.mjs`：`cosineSimilarity`、缓存校验、混合检索
- `server/app.mjs`：批量补齐、写后后台缓存、失败降级
- `server/model.test.mjs`：OpenAI-compatible Embedding 请求与返回顺序测试
- `server/memory.test.mjs`：无关键词语义召回和缓存失效测试
- `server/app.test.mjs`：语义检索、缓存落盘和失败降级集成测试

## 七、数据结构

```ts
interface MemoryItem {
  id: string;
  content: string;
  type: "user_preference" | "project_fact" | "conversation_summary";
  level: "high" | "medium" | "low";
  status: "active" | "candidate" | "disabled";
  op?: "add" | "update" | "disable" | "noop";
  targetId?: string;
  keywords?: string[];
  hash?: string;
  embedding?: number[];
  embeddingModel?: string;
  embeddingHash?: string;
  embeddingUpdatedAt?: string;
  accessCount?: number;
  lastAccessedAt?: string;
}
```

`embedding` 只保存在本地 `data/memory/index.json`，该目录已被 Git 忽略，不会上传到仓库。

## 九、后续观察项

- 用真实对话数据统计关键词命中率、语义召回率和误召回率，再调整 `0.35` 的默认语义阈值。
- 当活跃记忆超过数千条时，再评估向量分片或本地向量索引；当前几百条规模无需引入向量数据库。
- 对不同供应商记录 Embedding 模型兼容性，避免把聊天模型名误填为 Embedding 模型名。
