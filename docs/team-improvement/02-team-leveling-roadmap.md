# 团队技术能力提升路线图

> 目标：把"能跑起来"的团队，拉到"改得动、信得过、可维护"的水平。
> 原则：**少讲概念，多立规矩；少做审查，多做门禁；用工具把质量卡在合并前，而不是靠人盯人。**

---

## 一、现状评估（先对齐团队在哪个段位）

不要凭感觉。用下面 5 个维度给团队当前水平打分（1–5），这是后续所有动作的基线：

| 维度 | 现任评分（本项目实证） | 说明 |
|---|---|---|
| 代码规范一致性 | 2 | 中英混排、温度精度双端不一致、魔法字符串 |
| 安全素养 | 1 | CORS 全开无鉴权、模型请求无超时、prompt 注入 |
| 可测试性 | 1 | 零自动化测试 |
| 架构能力 | 2 | 1251 行单体、配置三处重复不一致 |
| 工程化（CI/门禁） | 1 | 无 lint 卡点、无测试门禁、无 PR 模板 |

**提升不是补知识，是先把这些"1 分"项用机制抬到 3 分。**

## 二、四大支柱（团队能力建设的骨架）

### 支柱 1：编码规范（让"正确"成为默认值）
- 引入 ESLint + Prettier，保存到时自动格式化，**CI 卡 lint 不过不让合**。
- 关键规则：禁止裸 `any`、禁止吞异常（catch 必须日志）、禁止魔法字符串、组件超 300 行必须拆。
- 建立项目级 `constants.ts` / `types.ts` **前后端共享**。
- 文案统一走 i18n，禁止界面硬编码英文标签。

### 支柱 2：代码评审流程（把经验沉淀进团队）
- 每个 PR 至少 1 名 reviewer，reviewer 必须对 **安全 + 测试** 两项明确打勾。
- 采用 `03-code-review-checklist.md` 作为 PR 模板的必填项。
- 评审关注点按层级：先正确性与安全 → 再可维护性 → 最后风格（风格交给 lint，别在评审里扯）。

### 支柱 3：测试文化（从"不敢改"到"放心改"）
- 任何新后端逻辑必须有单测（Vitest）。先覆盖 `generateCandidatesFromMessages`、`selectRelevantMemories`、`maskModelConfig`、`classifyProviderError`。
- 前端关键交互用 Playwright 做冒烟（你们已有 `scripts/capture-screenshots.mjs`，直接升级为断言式）。
- 目标：合并前测试必须全绿，覆盖率纳入看板（先不卡阈值，先看趋势）。

### 支柱 4：持续学习（让团队自己长能力）
- 每周 30 分钟"代码诊所"：拿一个真实 PR 或 bug 复盘，资深者带拆。
- 建立内部 `patterns/` 知识库（如"模型请求必须带超时""外部内容必须做角色隔离"）。
- 鼓励把踩过的坑沉淀成 Skill/文档，复用而非重复踩。

## 三、角色成长路径（不同段位练不同东西）

| 段位 | 重心 | 90 天可达目标 |
|---|---|---|
| 初级 | 规范执行 + 基础测试 | 写的代码 lint 0 错误、关键函数有单测、不引入魔法字符串 |
| 中级 | 模块拆分 + 安全意识 | 能把 1000 行组件拆干净、能识别 CORS/注入/超时类风险 |
| 高级 | 架构决策 + 评审把关 | 能定模块边界、能写评审门禁、能带代码诊所 |

## 五、度量指标（用数据看进步，不用体感）

- **合并前阻断率**：CI lint/测试未过却想合的次数 → 目标趋近 0。
- **缺陷逃逸率**：上线后由测试可覆盖的 bug 占比 → 逐月下降。
- **单测覆盖率趋势**：从 0 起步，看斜率。
- **严重问题存量**：C1/C2/M1/M2 类必须季度内清零。
- **PR 平均评审轮次**：反映规范是否真正内化为习惯。

## 六、资深开发者的角色定位

我会以 **"把关人 + 教练"** 双重身份介入：
- **把关**：对安全、架构、性能类改动做最后一道门禁（如本审计报告）。
- **教练**：不直接替团队写所有代码，而是把关键修复做成"带教样本"（如 C1/C2 我来写并讲解，其余由团队照模板补齐）。
