# Memory Agent 页面审查

日期：2026-07-10

## 审查范围

- 桌面三栏工作台与设置抽屉
- 1024px 桌面断点
- 800px 平板宽度
- 390px 手机聊天页与设置页
- 主要交互状态、响应式布局和可访问性风险

目标用户需要快速完成三件事：选择角色和会话、与智能体聊天、审核长期记忆。页面应始终把“继续对话”保持为最明显、最容易到达的主任务。

## 已确认优点

- 桌面三栏的职责明确：角色与会话、聊天、长期记忆彼此分区。
- 视觉语言稳定，白底、浅灰面和主蓝色适合高频工作台，不会抢占内容注意力。
- 消息已支持 Markdown，引用记忆可展开，长期记忆的“候选后确认”规则在界面上可感知。
- “整理记忆”可以一次点击开启、再次点击关闭，激活状态有视觉反馈和 `aria-pressed`。
- 抽屉使用统一进入/退出动效，并提供 `prefers-reduced-motion` 降级。

## P0 结构性问题

### 1. 手机输入区不在视口内

`.motion-page` 的入场动画保留了 `transform`，使内部 `position: fixed` 的 `.mobile-composer` 以整张 `.phone-frame` 为包含块。实测 390x844 时，输入区顶部位于约 2294px；用户进入聊天页后看不到输入框，需要滚过全部历史消息。

建议：

- 让 `.phone-frame` 使用 `height: 100dvh; overflow: hidden`。
- 让 `.mobile-chat-list` 使用 `min-height: 0; overflow-y: auto`。
- 不要在 `.mobile-composer` 的祖先上长期保留 `transform`；将页面入场动画放到不会包含 fixed 元素的内层。
- 输入区改为视口底部固定或容器内 sticky，并加入 `env(safe-area-inset-bottom)`。
- 首次加载直接定位到最后一条消息，后续新增消息再使用平滑滚动。

代码位置：`src/styles.css:1609`、`src/styles.css:1618`、`src/styles.css:1676`、`src/styles.css:1689`、`src/components/ChatPanel.tsx:180`。

### 2. 1024px 断点与三栏最小宽度冲突

桌面网格最小宽度为 `300 + 520 + 340 = 1160px`，但桌面布局从 1024px 就启用。实测 1024px 时 `scrollWidth=1160`，右侧记忆栏被裁掉 136px；快捷按钮也被压成两行。

建议：

- 将完整三栏断点提高到至少 1180px。
- 1024-1179px 使用紧凑工作台：默认收起右侧记忆栏，并把左栏缩到 248-264px。
- 768-1023px 不直接拉伸手机布局；使用最大宽度 520px 的居中单栏，或采用“会话列表抽屉 + 聊天”布局。
- 快捷操作在窄宽度下保留 2-3 个高频项，其余放入“更多”菜单，禁止固定高度按钮内换行。

代码位置：`src/styles.css:65`、`src/styles.css:74`、`src/styles.css:816`、`src/styles.css:1859`。

### 3. 配置输入每次按键都触发保存

角色标题、角色描述、Base URL、模型名和上下文长度的 `onChange` 会直接调用后端保存。用户输入一个完整值时会产生多次受保护请求，缺少管理员令牌时还会连续出现错误；并发响应也可能覆盖较新的输入。

建议：

- 设置表单改为本地 draft，底部提供明确的“保存更改”主按钮。
- 展示“未保存 / 保存中 / 已保存 / 保存失败”四种局部状态。
- 模型“测试连接”先校验并保存当前 draft，再测试该份配置。
- 若坚持自动保存，至少使用 500-800ms debounce，并防止旧请求覆盖新状态。

代码位置：`src/components/SettingsPanels.tsx:121`、`src/components/SettingsPanels.tsx:286`、`src/hooks/useWorkbenchState.ts:98`、`src/hooks/useWorkbenchState.ts:248`。

## P1 体验优化

### 4. 右侧记忆栏操作过密

每条记忆都常驻“编辑 / 禁用 / 删除”，五条内容同时出现时视觉噪声较高，也弱化了记忆内容本身。建议默认只展示类型、摘要和等级；行内保留一个菜单，完整管理放入 `memory.md` 抽屉。首屏显示最近或最高优先级的 3 条，并增加“查看全部”。

### 5. 设置页技术项优先级过高

手机设置首屏首先展示 `X-Admin-Token`，普通用户会先遇到系统概念。建议将“角色与行为”放在前面，把管理员令牌、Base URL、上下文长度归入“高级模型设置”，默认折叠。配置错误应在对应字段下说明，不只显示 `AUTH_REQUIRED` 代码。

### 6. 平板宽度缺少专门布局

800px 下消息气泡仍保持手机宽度，而顶部提示和输入区横跨页面，形成明显的宽窄失衡。建议给聊天主体设置 560-680px 的阅读列，外围居中；设置页可使用两列，但不要把手机卡片直接拉满。

### 7. 次要信息需要更人性化

- 右栏 `2478m` 应显示为“约 2 天前”或具体更新时间。
- 响应式 Web 不需要模拟 `9:41` 状态栏，容易与真实系统状态栏重复。
- 左侧会话和角色的删除按钮可以在 hover、选中或更多菜单中出现，降低视觉噪声和误触压力。
- 手机顶部说明“主页保持纯聊天……”是开发说明，不是任务内容；可改为一次性提示或移除。

## 可访问性风险

- 多个输入框使用 `outline: 0`，但没有统一 `:focus-visible` 替代样式，键盘用户难以判断焦点。
- 手机设置按钮、手机关闭按钮和发送按钮缺少可访问名称。
- 深度思考、联网搜索、工具等模式没有统一 `aria-pressed`；语言切换也没有 tab/pressed 语义。
- 抽屉虽然有 `role="dialog"` 和 `aria-modal="true"`，但没有 Escape 关闭、焦点约束和关闭后焦点归还。
- `#98a2b3` 在白底上的对比度约为 2.58:1，`#a4adba` 约为 2.27:1；用于 12-15px 状态文字和占位符时可读性不足。

建议先增加全局 `:focus-visible`，补齐 icon button 的 `aria-label`，统一模式按钮语义，再实现抽屉焦点管理。

## 建议执行顺序

1. 修复手机聊天容器和输入区定位。
2. 修正 1024px/平板断点与快捷操作布局。
3. 将角色和模型配置改为 draft + 明确保存。
4. 简化右侧记忆栏并重排手机设置优先级。
5. 补齐焦点、标签、模式状态、对比度和对话框键盘行为。

本次已验证页面加载、桌面设置抽屉、整理记忆开关、手机聊天/设置切换和响应式边界。没有发送真实模型请求、写入记忆或修改配置，以避免消耗 API 和改动本地数据。截图只能证明可见状态；完整屏幕阅读器体验、浏览器缩放 200% 和实体设备安全区仍需单独测试。
